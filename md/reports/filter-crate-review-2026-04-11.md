## 🔴 Critical (1)

### 1. `slug.rs:83` — UTF-8 truncation panics on multi-byte char boundary

`rslt[..max_length].to_string()` uses **byte indexing**. When Unicode characters
survive the regex (see High #1 below), truncation at a byte offset inside a
multi-byte character causes a **panic** — an unrecoverable crash.

**Reproduction:**

```rust
let filter = SlugFilter::new(8, true);
filter.filter(Cow::Borrowed("aaaaaaa世"));
// panics: "byte index 8 is not a char boundary; it is inside '世' (bytes 7..10)"
```

**Suggested fix:** Use `char_indices().nth(max_length)` to find a safe truncation
point:

```rust
if rslt.chars().count() > max_length {
    let byte_idx = rslt.char_indices().nth(max_length).map(|(i, _)| i).unwrap_or(rslt.len());
    Cow::Owned(rslt[..byte_idx].to_string())
} else {
    Cow::Owned(rslt)
}
```

## 🟠 High (2)

### 2. `slug.rs:7` — Regex allows Unicode word chars through, producing invalid slugs

The pattern `r"(?i)[^\w\-]"` uses Unicode-aware `\w` (the default in the `regex`
crate). This means CJK, Cyrillic, Arabic, and other Unicode word characters pass
through the filter **unmodified**, creating slugs that are not URL-safe.

**Reproduction:**

```rust
to_slug(Cow::Borrowed("hello世界world")) // → "hello世界world" (expected: "hello-world")
to_slug(Cow::Borrowed("你好"))            // → "你好"            (expected: "")
```

There is also an **internal inconsistency**: the fast-path function `is_valid_slug()`
(line 62) only allows ASCII characters (`c.is_ascii_lowercase() || c.is_ascii_digit()
|| c == '_'`), so Unicode slugs generated by the slow path would never match the fast
path.

**Suggested fix:** Use an ASCII-only character class:

```rust
static SLUG_FILTER_REGEX_STR: &str = r"(?i)[^a-zA-Z0-9_\-]";
```

Or disable Unicode mode: `r"(?-u)(?i)[^\w\-]"`.

### 3. `filter_op.rs:263` — Stack overflow on deeply nested `Chain`

`FilterOp::Chain(vec![FilterOp::Chain(vec![...])])` recurses through `apply_ref`.
At ~10,000 levels of nesting, this triggers a **stack overflow** — a process abort
that cannot be caught with `catch_unwind`.

Affects all three `Chain` implementations:
- **String** — `filter_op.rs:263–279`
- **Value** — `filter_op.rs:398–404`
- **Numeric** — `filter_op.rs:434–436`

**Reproduction:**

```rust
let mut chain = FilterOp::<String>::Trim;
for _ in 0..10_000 {
    chain = FilterOp::Chain(vec![chain]);
}
chain.apply("  hello  ".to_string()); // fatal: stack overflow, aborting
```

**Suggested fix:** Flatten nested chains iteratively before applying:

```rust
fn flatten(ops: &[FilterOp<T>]) -> Vec<&FilterOp<T>> {
    let mut flat = Vec::new();
    let mut stack = ops.iter().rev().collect::<Vec<_>>();
    while let Some(op) = stack.pop() {
        if let FilterOp::Chain(inner) = op {
            stack.extend(inner.iter().rev());
        } else {
            flat.push(op);
        }
    }
    flat
}
```

**Note:** This requires adversarial or programmatic input to trigger. It is not
exploitable through normal usage, but it is a soundness concern if filter
configurations are user-controlled or deserialized from untrusted input.

## 🟡 Medium (1)

### 4. `xml_entities.rs:85` — Capacity underestimate for entity-heavy strings

`String::with_capacity(input.len())` does not account for entity expansion. The
worst case is `&` → `&amp;` (5× expansion), so a string of all `&` characters would
cause multiple reallocations.

**Not a correctness bug** — output is always correct. Performance-only impact.

**Suggested fix:**

```rust
// Heuristic: assume ~20% of chars need encoding, with avg 4x expansion
String::with_capacity(input.len() + input.len() / 5 * 3)
```

## 🔵 Low (3)

### 5. `xml_entities.rs:54` / `slug.rs:73` — `Cow<'static, str>` output forces allocation on no-op

Both `XmlEntitiesFilter` and `SlugFilter` declare `type Output = Cow<'static, str>`.
This prevents returning a borrowed reference to the input when no transformation is
needed, forcing `into_owned()` even on the fast path.

**Correct but suboptimal.** Changing the output lifetime to match the input (`Cow<'a,
str>`) would eliminate unnecessary allocations on no-op inputs. This is a breaking API
change.

---

### 6. `try_filter_op.rs:156` — `TryFilterOp<Value>::Chain` missing empty-chain early return

The String implementation (line 120) has an optimization:

```rust
if ops.is_empty() {
    return Ok(Cow::Borrowed(value));
}
```

The Value implementation (line 155) lacks this and always clones the input, even for
an empty chain. Minor consistency issue with negligible performance impact.

---

### 7. `slug.rs:21` — Documentation claims incorrect pattern

The doc comment states the slug matches `/^\w[\w\-]{0,198}\w?$/`, but:

1. Unicode `\w` leak means actual output can contain non-ASCII word chars.
2. The pattern implies exactly 200 chars max (1 + 198 + 1), which is correct but
   misleading given the truncation uses byte length, not char count.
3. After fixing the Unicode issue, documentation should be updated to reflect ASCII-only
   output.
